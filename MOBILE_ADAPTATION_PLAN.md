# 移动端适配改造清单

## 背景

当前系统并不是完全没有移动端适配，但整体仍然明显偏桌面端设计。

现状特点：

- 登录、注册、Token 列表等基础页面已经有部分响应式处理。
- 核心业务页面仍以宽屏控制台布局为主。
- 很多高频页面在手机上虽然“能打开”，但并不“好用”。
- 响应式规则覆盖范围不够，且部分页面依赖大量 inline style，后续维护成本较高。

结论：

- 当前系统对手机并不友好。
- 主要问题集中在核心功能页，而不是全站完全不可用。
- 后续应优先做结构性移动端改造，而不是仅靠补几个 media query。

## 目标

移动端改造的目标不是简单缩放桌面布局，而是让高频功能在手机上具备可用性和可操作性。

目标包括：

- 高频页面支持单手操作和单列阅读。
- 减少横向拥挤、按钮挤压、表单过密的问题。
- 将复杂页面拆成更符合手机使用习惯的流程型界面。
- 建立统一的响应式规则，降低后续维护成本。

## 当前问题概览

### 1. 基础层问题较轻

- 全局样式中已有部分断点规则。
- `TokenImport`、`Login`、`Register` 等页面已有一定程度的移动端适配。
- 这些页面在手机上基本可用，但仍存在交互密度偏高的问题。

### 2. 核心功能层问题明显

- `GameStatus` 页面信息量大、卡片多、Tabs 多，不适合直接在小屏上平铺。
- `BatchDailyTasks` 页面本质上是桌面控制台，结构和交互复杂度都不适合手机。
- `Unlimitedlineup` 阵容助手依赖网格、拖拽、多弹窗，移动端交互体验较差。
- `ServerScheduledTasks` 表单和任务卡片仍偏桌面风格。

### 3. 交互层问题最突出

- 手机上更适合单列、分步、折叠式操作。
- 当前很多页面使用多按钮组、多列网格、并排信息块，不利于小屏操作。
- 某些弹窗宽度固定，不适合移动端视口。

## 改造优先级

建议按以下顺序推进：

1. `TokenImport`
2. `GameStatus`
3. `BatchDailyTasks`
4. `Unlimitedlineup`
5. `ServerScheduledTasks`

说明：

- `TokenImport` 是入口页，访问频率高，改造收益直接。
- `GameStatus` 是核心操作页，移动端信息密度问题明显。
- `BatchDailyTasks` 改造量最大，但也是最需要结构性重做的页面。

## 页面级改造清单

### 1. TokenImport

目标：

- 让 Token 导入、查看、选择、刷新等核心操作在手机上自然可用。

问题：

- 头部操作区横向控件过多。
- 排序按钮组占宽度大。
- 导入弹窗更接近桌面 modal，不适合手机。
- Token 卡片中的操作和信息较密集。

建议改造：

- 头部操作区改为单列堆叠布局。
- 排序方式从按钮组改为下拉选择器。
- 移动端导入弹窗改成全屏抽屉或底部面板。
- Token 列表在手机端默认使用列表模式，不优先展示卡片模式。
- 单个 Token 卡片中的次级操作收纳到“更多”菜单。
- 批量操作入口改成独立按钮或弹层，不与主操作并列铺开。

### 2. GameStatus

目标：

- 让日常、活动、工具、俱乐部等分区在手机上具备可读性和可操作性。

问题：

- 一级 Tabs 数量多。
- 二级 Tabs 也较多。
- 卡片数量多、内容密集，页面容易过长。
- 同时展示多个复杂组件，不利于小屏阅读。

建议改造：

- 一级 Tabs 改成可横向滚动的移动端导航。
- 每个分区默认只展开最核心内容，其余折叠。
- 二级 Tabs 改成分段器或折叠分组，减少同时暴露内容。
- 对 `daily`、`tools` 两个高频区优先做移动端精简布局。
- 对大卡片内部的并排指标改成单列展示。
- 为高频卡片提供移动端专用样式或专用轻量布局。

### 3. BatchDailyTasks

目标：

- 将桌面控制台式页面拆成适合手机的分步骤操作界面。

问题：

- 页面信息密度极高。
- 多区域同时展示，视觉压力大。
- 账号选择、任务设置、日志查看都在同一页面里。
- 不适合仅靠媒体查询压缩布局。

建议改造：

- 将页面重构为 3 个步骤：
  - 账号选择
  - 任务配置
  - 执行与日志
- 分组管理和排序操作改到抽屉或弹层。
- Token 列表改为单列卡片。
- 批量设置区域改成折叠面板。
- 定时任务列表和执行日志独立成单独视图或弹层。
- 减少页面中常驻的次级信息，按需展开。

### 4. Unlimitedlineup

目标：

- 让阵容查看、保存、应用、编辑在手机上可用。

问题：

- 英雄编辑区偏桌面网格布局。
- 拖拽排序不适合触屏。
- 已保存阵容列表信息较密集。
- 多个操作按钮横向排布。

建议改造：

- 阵容编辑区改成单列卡片。
- 拖拽排序保留桌面端，移动端补充“上移/下移”按钮。
- 已保存阵容弹窗改为全屏列表页或抽屉。
- 英雄信息分层显示，非核心信息折叠。
- 工具按钮改成纵向分组，而不是横向一排。

### 5. ServerScheduledTasks

目标：

- 让服务端定时任务配置页在手机上仍能顺畅操作。

问题：

- 任务卡片头部按钮过多。
- 新建/编辑任务表单偏桌面。
- 角色选择与任务设置区域密度较高。

建议改造：

- 任务卡片操作按钮收纳到“更多”菜单。
- 新建/编辑任务使用全屏表单页或全屏 modal。
- 角色选择改成可搜索单列列表。
- 任务设置从网格布局改成纵向分段布局。
- 日志查看使用全屏日志页，避免小屏 modal 体验差。

## 全局改造原则

### 1. 结构优先于样式

- 优先调整页面结构。
- 不要只依赖 media query 去压缩桌面布局。
- 对桌面控制台型页面，应考虑流程拆分。

### 2. 单列优先

- 手机端默认采用单列布局。
- 多列网格只保留在平板及以上尺寸。

### 3. 收纳次级操作

- 将批量操作、更多操作、次级设置放入抽屉、菜单或折叠面板。
- 保持主操作路径简洁明确。

### 4. 弹层移动端化

- 固定宽度 modal 应改为全屏或接近全屏。
- 避免 `40rem` 这类只适合桌面的固定尺寸。

### 5. 降低 inline style 依赖

- 将高频页面的 inline style 迁移为统一 class。
- 统一放入响应式样式文件或页面级样式中管理。

## 建议断点规则

建议统一以下断点：

- `<= 640px`
  - 手机单列模式
- `641px - 1024px`
  - 平板过渡模式
- `> 1024px`
  - 桌面模式

对应策略：

- 手机：单列、折叠、抽屉、全屏弹层
- 平板：保留部分双列，但减少密度
- 桌面：保留现有高密度布局

## 实施顺序建议

### 第一阶段

- 改造 `TokenImport`
- 建立通用移动端弹层和按钮组样式规则

### 第二阶段

- 改造 `GameStatus`
- 优先处理 `daily`、`tools` 两个高频分区

### 第三阶段

- 重构 `BatchDailyTasks`
- 将其从桌面控制台式布局改为移动端流程式布局

### 第四阶段

- 改造 `Unlimitedlineup`
- 改造 `ServerScheduledTasks`

## 验收标准

移动端改造完成后，至少应满足以下标准：

- 在常见手机宽度下不出现核心操作按钮被挤压不可点。
- 核心页面不依赖横向滚动完成主流程。
- 主流程操作不需要频繁缩放页面。
- 高频页面能够在手机上完成核心任务：
  - 导入 Token
  - 查看状态
  - 执行日常
  - 保存/应用阵容
  - 配置定时任务

## 结论

当前系统的移动端问题主要不在基础页面，而在核心业务页面。

后续应优先从以下三页开始：

1. `TokenImport`
2. `GameStatus`
3. `BatchDailyTasks`

这三页完成后，整体移动端可用性会明显提升。
